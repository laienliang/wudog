# 08 分阶段开发与联调验收计划

## 1. 总体原则

* 后续 Gemini CLI 每次只能执行一个阶段。
* 不得一次开发全部项目。
* 开发前必须先读取 `README.md`、`GEMINI.md` 和相关 docs 文档。
* 后端必须基于 Cool Admin Midway 当前实际版本，不按“从零创建普通 Midway 项目”推进。
* 每个阶段结束后都要输出阶段报告。
* 只有当前阶段验收通过后，才允许进入下一阶段。

## 2. 阶段计划

## 阶段 1：Cool Admin Midway 源码和版本只读检查

* 阶段目标：确认当前脚手架真实版本、目录规则、模块组织、基础实体、路由和配置方式。
* 前置条件：已阅读全部文档。
* 输入：`docs/02_系统总体架构设计.md`、`docs/10_Cool_Admin_Midway与课程技术栈适配说明.md`
* 具体任务：只读检查脚手架 `README`、`package.json`、示例模块、配置、基础实体、基础控制器和路由规则。
* 输出文件：检查记录、版本结论、差异说明。
* 验收标准：明确当前版本中管理员认证、CRUD、模块目录和路由接法。
* 常见风险：照搬旧版教程。
* 出错时的回退方式：停止编码，回到源码检查结论重新确认。
* 是否允许进入下一阶段：是，前提是只读结论清楚。

## 阶段 2：准备后端脚手架并验证原始项目启动

* 阶段目标：准备 Cool Admin Midway 后端工程并验证原始脚手架可真实启动。
* 前置条件：阶段 1 完成。
* 输入：阶段 1 检查结论。
* 具体任务：放置或准备后端脚手架、核对配置、真实启动默认项目并记录结果。
* 输出文件：后端脚手架目录、启动记录、问题清单。
* 验收标准：原始脚手架可真实运行，后端端口调整方案明确指向 `3000`。
* 常见风险：还未检查版本就直接改代码。
* 出错时的回退方式：回到原始脚手架状态，先修复环境或配置问题。
* 是否允许进入下一阶段：是，前提是原始脚手架启动已验证。

## 阶段 3：创建乌东非遗商品模块骨架

* 阶段目标：在 Cool 中建立第 1 组业务模块目录和基础文件骨架。
* 前置条件：阶段 2 完成。
* 输入：`docs/10_Cool_Admin_Midway与课程技术栈适配说明.md`
* 具体任务：创建独立业务模块，例如 `src/modules/wudong`，并规划 `controller/app`、`controller/admin`、`service`、`entity`。
* 输出文件：模块骨架文件、目录说明。
* 验收标准：模块边界清楚，未污染 Cool 现有管理员与权限基础系统。
* 常见风险：把业务逻辑散落到脚手架无关目录。
* 出错时的回退方式：保留最小模块骨架，撤回多余目录调整。
* 是否允许进入下一阶段：是。

## 阶段 4：商品分类和商品

* 阶段目标：完成商品分类和商品核心数据能力。
* 前置条件：阶段 3 完成。
* 输入：`docs/03_数据库设计.md`、`docs/04_后端API接口设计.md`
* 具体任务：实现分类和商品实体、管理 CRUD、公开列表和详情基础能力。
* 输出文件：实体、控制器、服务、DTO、测试记录。
* 验收标准：库存未进入商品表；商品详情包含工艺介绍和传承人信息。
* 常见风险：遗漏上架状态字段或详情字段。
* 出错时的回退方式：回退到能稳定返回分类和商品详情的最小实现。
* 是否允许进入下一阶段：是，前提是核心商品链路可用。

## 阶段 5：SKU 和商品图片

* 阶段目标：完成 SKU 和商品图片能力。
* 前置条件：阶段 4 完成。
* 输入：`docs/03_数据库设计.md`、`docs/04_后端API接口设计.md`
* 具体任务：实现 SKU 管理、库存修改、商品图片管理和排序。
* 输出文件：实体、控制器、服务、DTO、测试记录。
* 验收标准：库存不为负；售罄状态计算正确；图片排序可用。
* 常见风险：库存写回商品表或遗漏 SKU 状态。
* 出错时的回退方式：回退到 SKU 与图片基础查询可用的状态重新补校验。
* 是否允许进入下一阶段：是。

## 阶段 6：普通用户商品接口

* 阶段目标：完成普通用户公开商品接口。
* 前置条件：阶段 5 完成。
* 输入：`docs/04_后端API接口设计.md`
* 具体任务：实现公开分类、商品列表、商品详情、SKU 和图片接口。
* 输出文件：`controller/app`、服务、测试记录。
* 验收标准：普通用户只能看到已上架且未删除商品。
* 常见风险：公开接口泄露下架商品。
* 出错时的回退方式：优先恢复状态过滤和软删除过滤。
* 是否允许进入下一阶段：是。

## 阶段 7：收藏接口

* 阶段目标：完成收藏、取消收藏、收藏状态和我的收藏。
* 前置条件：阶段 6 完成。
* 输入：`docs/03_数据库设计.md`、`docs/04_后端API接口设计.md`
* 具体任务：实现收藏相关接口和去重逻辑。
* 输出文件：实体、控制器、服务、测试记录。
* 验收标准：数据库唯一约束与业务恢复逻辑同时成立。
* 常见风险：只依赖 Redis 去重，或重复插入收藏关系。
* 出错时的回退方式：先恢复数据库唯一约束，再修正服务逻辑。
* 是否允许进入下一阶段：是。

## 阶段 8：管理接口和商品上下架

* 阶段目标：完成后台管理接口和商品上下架。
* 前置条件：阶段 7 完成。
* 输入：`docs/04_后端API接口设计.md`、`docs/10_Cool_Admin_Midway与课程技术栈适配说明.md`
* 具体任务：复用 Cool 管理认证体系，实现后台分类、商品、SKU、图片和上下架接口。
* 输出文件：`controller/admin`、服务、测试记录。
* 验收标准：不重复设计管理员与权限系统；商品上架前校验完整。
* 常见风险：绕开 Cool 现有认证体系重新写管理员系统。
* 出错时的回退方式：回到 Cool 现有认证能力，再补业务接口。
* 是否允许进入下一阶段：是。

## 阶段 9：Swagger 与 Postman 测试

* 阶段目标：生成真实 Swagger / OpenAPI，并完成 Postman 核心接口测试。
* 前置条件：阶段 8 完成。
* 输入：`docs/04_后端API接口设计.md`
* 具体任务：配置 Swagger、整理 Postman、执行真实接口测试并记录截图。
* 输出文件：Swagger 接口文档、Postman 集合、测试截图、测试报告。
* 验收标准：Swagger 与真实接口一致；Postman 可验证核心链路。
* 常见风险：只有 Swagger，没有真实联调。
* 出错时的回退方式：以 Postman 和真实数据库联调结果为准重新核对。
* 是否允许进入下一阶段：是。

## 阶段 10：React PC Web

* 阶段目标：完成 React PC Web 列表、详情、新增、编辑页面。
* 前置条件：阶段 9 完成。
* 输入：`docs/05_PC_Web端页面与交互设计.md`
* 具体任务：实现 React 页面、路由、请求封装和状态处理。
* 输出文件：PC Web 页面文件、路由、请求层。
* 验收标准：页面真实连接后端；不改为 Vue。
* 常见风险：误接后台权限接口或引入错误前端框架。
* 出错时的回退方式：先恢复只读商品页，再补表单联调。
* 是否允许进入下一阶段：是。

## 阶段 11：原生微信小程序

* 阶段目标：完成原生小程序列表、详情、我的收藏页面。
* 前置条件：阶段 9 完成。
* 输入：`docs/06_微信小程序端页面与交互设计.md`
* 具体任务：实现原生小程序页面、请求层和收藏交互。
* 输出文件：小程序页面文件、配置、请求工具。
* 验收标准：开发者工具中可真实联调；不改为 UniApp 或 Taro。
* 常见风险：真机直接请求电脑 `localhost`。
* 出错时的回退方式：回到开发者工具本地调试验证链路。
* 是否允许进入下一阶段：是。

## 阶段 12：React 管理后台

* 阶段目标：完成 React + Ant Design 管理后台。
* 前置条件：阶段 9 完成。
* 输入：`docs/07_管理后台页面与运营功能设计.md`
* 具体任务：实现登录页、布局页、商品管理、分类管理、SKU 管理、图片管理。
* 输出文件：后台页面、路由、请求拦截器、权限处理。
* 验收标准：后台真实联调；不改为 cool-admin-vue。
* 常见风险：把后端脚手架误当成前端模板。
* 出错时的回退方式：先修复登录态和路由保护，再恢复业务页。
* 是否允许进入下一阶段：是。

## 阶段 13：四端联调

* 阶段目标：完成后端、PC Web、小程序、管理后台四端联调。
* 前置条件：阶段 10、11、12 完成。
* 输入：全部设计文档与已实现代码。
* 具体任务：验证浏览、收藏、上下架、库存变化在各端一致。
* 输出文件：联调记录、问题清单、修复说明。
* 验收标准：商品上下架、收藏状态、售罄状态在四端表现一致。
* 常见风险：字段命名或接口约定不一致。
* 出错时的回退方式：锁定单条链路逐端排查。
* 是否允许进入下一阶段：是。

## 阶段 14：`init.sql`、截图和验收资料

* 阶段目标：整理课程验收所需 SQL、截图和说明材料。
* 前置条件：阶段 13 完成。
* 输入：数据库实体、联调结果、测试截图。
* 具体任务：生成并核对 `backend/sql/init.sql`，确保与 Entity 同步；整理截图和验收资料。
* 输出文件：`init.sql`、截图、验收说明。
* 验收标准：SQL 与 Entity 不冲突，资料齐全。
* 常见风险：出现两套不一致的数据结构。
* 出错时的回退方式：回到实体重新导出并逐项核对。
* 是否允许进入下一阶段：是。

## 阶段 15：Git 整理和代码审查

* 阶段目标：整理 Git 变更、协作记录并完成代码审查。
* 前置条件：阶段 14 完成。
* 输入：全部实现代码与资料。
* 具体任务：整理提交范围、检查无关文件、执行代码审查或自查。
* 输出文件：Git 整理记录、代码审查记录。
* 验收标准：有真实 Git 提交记录和审查结论。
* 常见风险：误删他人文件或跳过审查。
* 出错时的回退方式：按文件粒度恢复误改，再重新整理。
* 是否允许进入下一阶段：是。

## 阶段 16：Docker 容器化部署

* 阶段目标：补齐课程整体技术栈中的 Docker 部署能力。
* 前置条件：阶段 15 完成。
* 输入：`docs/10_Cool_Admin_Midway与课程技术栈适配说明.md`
* 具体任务：规划并实现后端 Dockerfile、MySQL、Redis、Docker Compose、环境变量和网络联通。
* 输出文件：Dockerfile、Compose 配置、部署说明。
* 验收标准：容器可真实启动并验证接口联通。
* 常见风险：在早期阶段过早引入容器复杂度。
* 出错时的回退方式：保留本地直连可运行方案，再逐步恢复容器化。
* 是否允许进入下一阶段：是。

## 阶段 17：8 分钟演示准备

* 阶段目标：完成 8 分钟现场演示材料和讲解顺序。
* 前置条件：阶段 16 完成。
* 输入：已联调系统、测试资料、部署资料。
* 具体任务：整理演示脚本、关键路径、备用截图和故障预案。
* 输出文件：演示提纲、演示顺序说明、备用截图。
* 验收标准：可在 8 分钟内清楚展示第 1 组核心能力与课程技术栈。
* 常见风险：演示链路过长、依赖外部网络或环境不稳定。
* 出错时的回退方式：缩短演示链路，优先展示商品、收藏、上下架主链路。
* 是否允许进入下一阶段：本阶段为最终准备阶段。

## 3. 阶段执行纪律

* 每次只开一个阶段任务。
* 若上一阶段未真实完成，不得跳过。
* 若当前阶段存在阻塞，应先输出阻塞报告，而不是强行继续。
