## 4. Cool Admin Midway 开发前置要求

* 后端必须基于 Cool Admin Midway 当前实际版本。
* 开发前必须只读检查脚手架 `README`、`package.json`、示例模块、配置、基础实体、基础控制器和路由规则。
* 管理员登录、JWT、角色权限、菜单、统一响应和基础 CRUD 优先复用 Cool 已有能力。
* 不得重复设计一套管理员、JWT 和权限基础系统。
* 普通用户接口与管理接口分开规划，例如 `controller/app` 和 `controller/admin`。
* `@CoolController` 和自动 CRUD 可用于普通增删改查，但不能代替库存、上架、收藏去重、软删除过滤和售罄计算等业务校验。

## 5. 接口规范

* 统一前缀：`/api`
* 路径命名：全小写 + 连字符
* 响应格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

* 分页结构：`total`、`page`、`pageSize`、`list`
* 公开商品接口仅返回已上架且未删除商品
* 收藏、写操作和后台管理接口必须使用 JWT Bearer Token
* Cool 自动路由不一定天然符合课程接口路径，正式开发时必须通过路由前缀或自定义 Controller 做兼容

## 6. 数据库规范

* 核心业务表：`product_category`、`product`、`product_sku`、`product_image`、`favorite`
* 普通用户最小支撑表：`user`
* Cool 管理员、角色、权限、菜单基础表复用当前脚手架版本，不在第 1 组单独重建
* 公共字段：`id`、`created_at`、`updated_at`、`is_deleted`
* 库存只允许放在 `product_sku.stock`
* 收藏去重依赖 `favorite(user_id, product_id)` 唯一关系
* 商品详情必须保留工艺介绍与传承人信息
* `backend/sql/init.sql` 最终必须与 Entity 保持同步，不允许两套冲突结构

## 7. 安全规则

* 管理员接口统一复用 Cool 管理员 JWT 与权限体系
* 收藏接口统一校验普通用户 JWT
* 不信任前端传来的上下架、库存和权限状态
* SKU 库存不得为负数
* 所有删除默认采用软删除

## 10. Redis、Swagger、Docker 定位

### 10.1 Redis

* 可用于商品缓存、Token 辅助管理、限流和防重复操作
* 核心商品 CRUD 必须在 Redis 未接入时也能运行
* 收藏去重必须由数据库唯一约束兜底

### 10.2 Swagger / OpenAPI

* 后端后续应生成真实接口文档
* Swagger 不能代替 Postman 和真实数据库联调

### 10.3 Docker

* Docker 是课程整体技术栈和后期部署目标
* 当前不是文档阶段任务
* 当前不需要在 Docker 中安装 Codex、Gemini CLI 或 Claude Code

## 13. 如何验证真实运行结果

* 真实执行命令并记录终端输出
* 真实启动服务并验证端口
* 真实发起接口请求并保存响应
* 真实打开页面并记录页面行为
* 若某项未执行，不得写成“已完成”

## 14. 如何避免假数据冒充联调成功

* 不使用前端本地常量数组代替后端接口成功
* 不用截图伪装实际可运行页面
* 收藏、上下架、库存变化必须验证真实后端响应与数据库变化

## 15. 如何避免一次修改过多文件

* 每次只围绕一个阶段修改文件
* 每次优先修改与阶段直接相关的目录
* 若改动面扩大，先暂停并重新核对阶段边界

## 16. 如何避免覆盖已有内容

* 修改前先读取目标文件
* 优先增量编辑，不整体重写
* 如果发现现有实现与文档冲突，先记录，再进行最小修正

## 17. 如何处理命令执行失败

* 原样记录失败命令与错误信息
* 分析失败是否由环境、权限、依赖或代码导致
* 若不能安全继续，输出阻塞说明并停止当前阶段

## 18. 如何记录未解决问题

* 按“问题描述 / 影响范围 / 当前判断 / 建议下一步”记录
* 未解决问题必须出现在阶段报告中，不得静默忽略
